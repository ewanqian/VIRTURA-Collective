# VIRTURA 日历编排框架 2026-03-16

> 注：这份文档保留为内部过程稿。当前对外口径以 [未来 8 周活动路线图](./eight-week-activity-roadmap-2026-03-16.md) 和 [Activities / Planned](../activities/planned/README.md) 为准。旧文中的 `Practice`、`Concept Briefing` 等命名已停用，对外统一改为 `在制内容审计会`、`Digital Library / Digital Stage 公开说明` 等更直接的说法。

这份文档回答的是一个更实际的问题：

`lu.ma/VIRTURA` 以后到底该发什么，不该发什么，以及怎么让它和 GitHub 里的作品、研究、站点和档案形成自然衔接。

它不是给外部读者看的品牌文案，而是给团队内部做节目单和日历管理用的工作说明。

---

## 一、先把角色定清楚

### 1. `lu.ma/VIRTURA`

负责：

- 长期订阅
- 公开提醒
- 报名与申请
- 开幕时刻
- 公开测试和观看窗口

它不只是“活动列表”，而应该是你们最稳定的公共时间入口。

### 2. `VIRTURA-Collective`

负责：

- 团队主入口
- 作品、活动、合作网络的第一层介绍
- 让第一次来的人知道自己该往哪走

### 3. `VIRTURA-SpacePort`

负责：

- 知识网络
- 项目档案
- 活动站点
- `MOVA` 线上展映
- 更深一层的资料和长期阅读

---

## 二、什么值得发到日历里

不是只有“线下活动”才值得进 Luma。

对 VIRTURA 来说，只要满足以下任一条件，就可以成为一个日历节点：

- 有明确的开始时间
- 有明确的申请窗口
- 有明确的观看窗口
- 有明确的测试时段
- 有明确的开幕或上线时刻

也就是说，这些都可以发：

- 线下茶话会
- 线上分享
- 公开测试
- Alpha / Beta
- 在制内容审计会
- 围绕 `Digital Library / Digital Stage` 的公开说明
- 页面或场景开放
- `MOVA` 线上展映开启

反过来，如果只是静默更新一个页面，没有开始时间，也没有公共动作，就不必强行发成日历节点。

## 三、建议保留的 7 种日历节点

### 1. Space Salon / 数字沙龙 / 交流客厅

适合：

- 作品分享
- 嘉宾漫谈
- 方法传播
- 轻量工作坊

适合的标题风格：

- 一个问题
- 一个主题
- 一个阶段性分享

### 2. 太空气球 / Baloon Live Space

适合：

- 线上演出空间测试
- 音画实验
- 共测夜
- 音乐人 / 视觉创作者协作节点

适合的标题风格：

- test
- session
- open rehearsal
- live experiment

### 3. Practice / 开放实践

适合：

- 共练
- 开放排练
- 带设备一起测试
- 带半成品来一起看

这个类型的好处是准备压力低，而且最容易把“只是关注的人”慢慢转成真正参与的人。

### 4. Concept Briefing / 概念短讲

适合：

- 介绍一个核心概念
- 介绍一个研究切口
- 把一页文档转成一次 20-40 分钟的公共说明

例如：

- 什么是数字情感容器
- 为什么我们把活动日历放这么前
- 为什么知识网络不能只是目录

### 5. Public Test / Alpha / Beta

适合：

- 新空间
- 新交互
- 新观看入口
- 新发布结构

重点不是把它包装得很成熟，而是明确告诉别人：我们在测试什么，需要什么反馈。

### 6. Release / Archive Opening

适合：

- 新页面上线
- 新场景开放
- 新一轮档案公开
- 作品观看入口开放

这个类型适合和 GitHub 的更新同步，但前提是它真的有一个“公开开始时刻”。

### 7. MOVA Online Exhibition / Screening

适合：

- 线上展映
- 作品观看室
- library 转展览
- 项目版本观看路线

这个类型不一定需要完整直播，更像“线上展览开幕”。

## 四、建议的活动与发布线分工

| 线 | 更适合承接什么 | 主要宿主 |
|---|---|---|
| `Space Salon` | 分享、漫谈、方法传播、嘉宾交流 | Luma + `stations/space-salon` |
| `太空气球` | 线上演出空间测试、音画实验、公开测试 | Luma + 相关项目页 |
| `Practice` | 共练、排练、半成品反馈、设备体验 | Luma + `stations` |
| `Concept Briefing` | 核心概念短讲、研究切口说明 | Luma + `research` / `knowledge-network` |
| `Release` | 页面开放、场景开放、档案上新 | Luma + GitHub 对应页面 |
| `MOVA` | 线上展映、作品观看室、主题小展 | Luma + `stations/mova` |

## 五、日历页里最值得明确写出来的事

如果 Luma 公开页还有地方继续写，最应该强调的是这三点：

1. 这里是长期订阅入口，不只是偶发活动列表
2. 这里会发布的不只是沙龙，还包括测试、practice、公开发布和线上展映
3. 如果想继续深读，活动之后可以顺着 GitHub 入口往下看

## 六、一个月的建议排法

如果按一个月 4 周来运营，比较稳的结构可以是：

### 第 1 周

- `Space Salon`
- 作用：建立认知

### 第 2 周

- `Practice` 或 `Public Test`
- 作用：拉近参与距离，收集反馈

### 第 3 周

- `Concept Briefing` 或 `方法短讲`
- 作用：把研究和知识网络公开化

### 第 4 周

- `MOVA` 或 `Release`
- 作用：把积累的页面、作品、档案转成可观看的公共节点

这个节奏的好处是：

- 不会每周都陷入高组织成本
- 活动和发布能互相支撑
- GitHub 不是只被动当资料库，而会定期被转成新的公共入口

## 七、第一批建议放进日历的节点类型

如果按你们当前的成熟度，最适合先稳定做的，是下面这 8 种：

1. `Space Salon` 主题分享
2. `太空气球` Alpha / Beta 共测
3. `Practice Night` 半成品共看
4. `Concept Brief` 数字情感容器
5. `Concept Brief` 知识网络怎么做成能用的
6. `Release` 本周开放一个新的页面或房间
7. `MOVA` 线上展映开启
8. `Archive Opening` 某个项目版本观看入口开放

## 八、每类节点建议配套的 GitHub 页面

| 节点类型 | 最少需要配套什么 |
|---|---|
| `Space Salon` | 一期活动的 README 或 public-release |
| `太空气球` | 测试说明、目标、反馈方式 |
| `Practice` | 一页简单的 practice 说明或相关主题入口 |
| `Concept Briefing` | 一条研究页或知识网络深读链接 |
| `Public Test` | 测什么、谁适合来、反馈怎么给 |
| `Release` | 一个真实可访问的新页面 |
| `MOVA` | 一页线上展映入口或观看路线 |

## 九、当前最值得立刻加进日历逻辑里的第三条线

就是 `MOVA / Museum of Virtual Art`。

原因很简单：

- 它能承接“不是见面活动、但值得公开发布”的节点
- 它能把 GitHub 里的 library 直接转成更容易传播的展示入口
- 它比硬做一次大型线上发布会轻得多
- 它很适合你们现在“内容已经有了，但还在整理前台”的阶段

所以对现在的 VIRTURA 来说，`MOVA` 不是一个可有可无的概念页，而是一条非常实用的发布线。

## 十、一个实用判断标准

如果团队在犹豫一个东西该不该发到日历里，可以只问两个问题：

1. 这个节点是否有明确的公共开始时刻？
2. 这个节点发出去后，外部读者是否有事可做？

如果两个答案都是“有”，它就值得进日历。

外部读者能做的事包括：

- 报名
- 订阅
- 观看
- 申请测试
- 继续深读
- 转发给朋友

如果这几件事都做不到，那它更适合先留在 GitHub 里继续整理，而不是急着发到 Luma。
