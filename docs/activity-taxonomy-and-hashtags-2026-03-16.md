# Activity Taxonomy And Hashtags 2026-03-16

这份文档解决 4 件事：

- 给 `VIRTURA` 现在这套活动一个统一分类
- 给 `Luma / GitHub / 小红书` 提供一套能复用的 hashtags
- 给后面轻量 side-ops 找到固定发布容器
- 避免每次发活动时都重新想“这到底属于什么”

这不是一套很花的运营词表，而是一套能继续用的工作表。

---

## 一、先记住这 6 个活动类别

`VIRTURA` 现在最稳定的活动结构，就是下面 6 类。

| 类别 | 中文说明 | 适合的语气 | 主要宿主 |
|---|---|---|---|
| `Space Salon` | 数字沙龙 / 交流客厅 / 嘉宾分享 / 一起放东西看 | 亲切、邀请、带问题来聊 | `Collective / activities` + `SpacePort / stations/space-salon` |
| `Baloon Live Space` | 线上数字舞台 / live set / 视觉共演 / 共测夜 | 现场感、测试感、共演感 | `SpacePort / stations/baloon-live-space` |
| `MOVA` | 线上展映 / 观看室 / 主题小展 / 发布节点 | 观看、展映、路线感 | `SpacePort / stations/mova` |
| `内容审计会` | 在制项目批评 / 页面复盘 / 版本审计 / 结构复盘 | 直接、诚实、欢迎挑问题 | `SpacePort / stations/content-audit` |
| `Digital Library / Digital Stage` | 仓库、归档、知识网络、前台和 AI 工作流公开说明 | 解释、拆解、讲清方法 | `SpacePort / stations/digital-library-stage` |
| `发布与观看入口开放` | 页面上新 / 观看室开放 / 新入口接入日历 | 简短、明确、告诉人去哪 | `SpacePort / stations/mova` 或对应系列页 |

一句话记法：

- 想聊人和过程，走 `Space Salon`
- 想看线上演出空间，走 `Baloon Live Space`
- 想看作品观看路线，走 `MOVA`
- 想认真挑问题，走 `内容审计会`
- 想听方法和结构，走 `Digital Library / Digital Stage`
- 想跟上最新上线入口，走 `发布与观看入口开放`

## 二、Luma 这边怎么理解“分类”和“精选”

我核过当前 `Luma` 后台，公开页现在已经会显示这几个发现标签：

- `人工智能`
- `艺术与文化`
- `科技`

当前标准后台里，能直接看到的是“已列入哪些精选分类页”，以及“取消列出”；没有一个非常明确、独立的手动下拉框，让你每场活动重新自定义一个新分类。

所以对现在这套系统来说，更有效的办法不是强行发明后台里没有的字段，而是把下面 4 件事稳定住：

1. 系列名字要稳定
2. 活动标题要直接
3. 活动详情第一段要说清楚发生什么
4. GitHub 入口和活动前阅读要持续补全

对现在这批活动，建议维持的发现方向是：

| 系列 | 优先落点 |
|---|---|
| `Space Salon` | `艺术与文化` / `科技` |
| `Baloon Live Space` | `科技` / `艺术与文化` |
| `MOVA` | `艺术与文化` |
| `内容审计会` | `科技` / `人工智能` |
| `Digital Library / Digital Stage` | `人工智能` / `科技` |
| `发布与观看入口开放` | 跟着所挂靠的主系列走 |

- `2-3` 个系列固定标签

- `1-2` 个城市或形式标签

## 五、适合轻量发布的 side-ops 题库

这些不是“大活动”，更像适合插在活动之间的小节点。好处是：

- 成本低
- 更新快
- 能持续给日历和仓库输血
- 也更适合转成 blog、小红书或一页研究卡片

### A. 一页说明型

- `为什么很多知识网络最后只剩目录`
- `作品越做越多，首页为什么反而更难看`
- `一个活动结束后，哪些资料最值得先归档`
- `为什么不是每个项目都需要单独做长文`
- `从一次公开说明里，怎么拆出 3 篇更轻的文章`

### B. 工具观察型

- `最近我们怎么用 GitHub 管活动、研究和项目档案`
- `AI 在填表、起草、补结构上到底省了哪些时间`
- `PlayCanvas 现在到底适不适合做轻量公开前台`
- `浏览器三维舞台和 VRChat 地图，各自卡在哪`
- `一次活动页里，哪些链接真的有人点`

### C. 档案开放型

- `Drop Flow 某个旧版本重新打开看，会发现什么`
- `Timer 某次现场为什么后来不再沿用`
- `一次失败的网页结构，后来是怎么收回去的`
- `一个还没公开的扫描测试，为什么先只做档案`
- `早期 Space Salon 文案，现在回看哪里写得太满`

### D. 社群互动型

- `带一个最近反复在看的网页来`
- `带一段最近没讲顺的视频来`
- `带一个最近最想吐槽的页面来`
- `带一个你舍不得删的旧版本来`
- `带一个你最近最想补但一直没补的条目来`

### E. 发布前小节点

- `下周这场活动前先看哪 3 个页面`
- `这次展映如果只能先看 10 分钟，建议怎么走`
- `本周回放和反馈整理出来了什么`
- `这轮测试到底修掉了哪 3 个问题`
- `下一次共测夜我们最想收到什么反馈`

## 六、推荐的发布层级

为了防止每次都把内容发得很重，建议后续固定分成 3 层：

### 第 1 层：日历节点

只负责：

- 这是什么
- 什么时候
- 为什么值得来
- 先看哪两篇

### 第 2 层：GitHub 档案页

负责：

- 完整背景
- 过程说明
- 前因后果
- 关联页面

### 第 3 层：研究长文 / blog

负责：

- 方法和判断
- 较完整的论证
- 后续还能被引用和拆分的内容

`Digital Library / Digital Stage` 这条线尤其适合这样做，因为它天然会同时产出：

- 一场活动
- 一篇长文
- 一份档案
- 几条轻量 side-ops

---

## 七、最值得继续补的 3 件事

1. 给每个活动系列补一张固定封面或视觉模板
2. 活动后 48 小时内补一页“回放 / 反馈 / 后续阅读”
3. 从 `Digital Library / Digital Stage` 和 `Baloon Live Space` 里持续拆出轻量 blog

只要这 3 件事稳定下来，`Luma`、`GitHub` 和后续 blog 就会自然接成一套，而不是每次都像重新搭台。
